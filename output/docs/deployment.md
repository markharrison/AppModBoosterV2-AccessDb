# Deployment Guide - Northwind Traders

## Prerequisites

| Requirement                 | Purpose                                                                |
| --------------------------- | ---------------------------------------------------------------------- |
| Azure subscription          | Host all cloud resources                                               |
| GitHub repository           | Source code + Actions workflows                                        |
| Azure CLI (`az`)            | Login, OIDC setup, Bicep deploy                                        |
| Azure Bicep CLI             | Included with Azure CLI 2.20+; verify with `az bicep version`          |
| .NET 10 SDK                 | Build and publish application                                          |
| PowerShell 7+               | Run setup scripts (`setup-identity.ps1`, `setup-sql-identity.ps1`)     |
| SqlServer PowerShell module | Required by `setup-sql-identity.ps1` (auto-installed if missing)       |
| GitHub CLI (`gh`)           | Set GitHub secrets automatically from `setup-identity.ps1`             |
| Azure AD permissions        | Application Administrator + User Access Administrator (for OIDC setup) |

## Infrastructure Resources

The following Azure resources are provisioned per environment (`dev`, `staging`, `prod`):

| Resource                | Name                      | Notes                                                       |
| ----------------------- | ------------------------- | ----------------------------------------------------------- |
| Resource Group          | `rg-accessdbapp-{env}`    | Container for all env resources; created by workflow        |
| App Service Plan        | `plan-accessdbapp-{env}`  | Linux B1, shared by Web App and API App                     |
| API App                 | `api-accessdbapp-{env}`   | ASP.NET Core, system-assigned MI, HTTPS-only, TLS 1.2       |
| Web App                 | `app-accessdbapp-{env}`   | ASP.NET Core Razor, system-assigned MI, HTTPS-only, TLS 1.2 |
| SQL Server              | `sql-accessdbapp-{env}`   | Entra-only auth, no SQL passwords, TLS 1.2                  |
| SQL Database            | `sqldb-accessdbapp-{env}` | Basic (5 DTU, 2 GB), SQL_Latin1_General_CP1_CI_AS           |
| Application Insights    | `appi-accessdbapp-{env}`  | Workspace-based, wired to both apps                         |
| Log Analytics Workspace | `log-accessdbapp-{env}`   | PerGB2018, 30-day retention                                 |

All resources are defined in `output/infra/main.bicep` and its modules under `output/infra/modules/`.

## GitHub Environment Setup

Three GitHub environments must exist in the repository before running any workflows: `dev`, `staging`, `prod`.

Each environment requires the following secrets (set automatically by `setup-identity.ps1`):

| Secret Name             | Description                                     |
| ----------------------- | ----------------------------------------------- |
| `AZURE_CLIENT_ID`       | App registration client ID for this environment |
| `AZURE_TENANT_ID`       | Azure AD tenant ID                              |
| `AZURE_SUBSCRIPTION_ID` | Azure subscription ID                           |

**To create environments manually**: GitHub repo -> Settings -> Environments -> New environment.

## Step-by-Step Deployment

### Step 1: Configure OIDC Identity

Run `setup-identity.ps1` once to create app registrations, OIDC federated credentials, and set GitHub secrets for all environments.

```powershell
# Login to Azure and GitHub first
az login
gh auth login

# Run the setup script
.\output\scripts\setup-identity.ps1 `
  -GithubOrg   "your-github-org" `
  -GithubRepo  "your-repo-name" `
  -SubscriptionId "00000000-0000-0000-0000-000000000000"
```

The script:

- Creates one app registration per environment (`sp-accessdbapp-dev`, `sp-accessdbapp-staging`, `sp-accessdbapp-prod`)
- Configures OIDC federated credentials scoped to each GitHub environment
- Assigns Contributor on the subscription
- Sets `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, and `AZURE_SUBSCRIPTION_ID` secrets in GitHub

The script is idempotent - safe to rerun.

### Step 2: Prepare Bicep Parameter Files

Before deploying, fill in the Azure AD group values in each environment's parameter file:

| File                                         | Value to fill                                |
| -------------------------------------------- | -------------------------------------------- |
| `output/infra/parameters/dev.bicepparam`     | `sqlAdminGroupObjectId`, `sqlAdminGroupName` |
| `output/infra/parameters/staging.bicepparam` | `sqlAdminGroupObjectId`, `sqlAdminGroupName` |
| `output/infra/parameters/prod.bicepparam`    | `sqlAdminGroupObjectId`, `sqlAdminGroupName` |

To find your Azure AD group object ID:

```bash
az ad group show --group "your-group-name" --query id -o tsv
```

These values can also be supplied at workflow runtime via the `deploy-infra.yml` workflow inputs (recommended for first run).

### Step 3: Deploy Infrastructure

Trigger the **Deploy Infrastructure** workflow from GitHub Actions:

1. Go to **Actions** -> **Deploy Infrastructure** -> **Run workflow**
2. Choose the target environment (`dev`, `staging`, or `prod`)
3. Enter the `sqlAdminGroupObjectId` and `sqlAdminGroupName` inputs
4. Click **Run workflow**

The workflow:

- Creates the resource group `rg-accessdbapp-{env}` in `eastus`
- Deploys `output/infra/main.bicep` with environment-specific parameters
- Provisions all 8 resources listed above

Alternatively, deploy from the CLI:

```bash
az group create --name rg-accessdbapp-dev --location eastus
az deployment group create \
  --resource-group rg-accessdbapp-dev \
  --template-file output/infra/main.bicep \
  --parameters output/infra/parameters/dev.bicepparam \
  --parameters sqlAdminGroupObjectId="<object-id>" \
  --parameters sqlAdminGroupName="<group-name>"
```

### Step 4: Configure SQL Identity

After infrastructure is deployed, run `setup-sql-identity.ps1` to create the contained database user for the API App's managed identity and grant the required permissions.

```powershell
# Login as a user with db_owner or higher on the target database
az login

.\output\scripts\setup-sql-identity.ps1 -Environment dev
```

The script:

- Installs the `SqlServer` PowerShell module if not present
- Acquires an access token for Azure SQL via `az account get-access-token`
- Creates a contained user for the API App managed identity (`api-accessdbapp-{env}`)
- Grants `db_datareader`, `db_datawriter`, and `db_ddladmin` roles

The script is idempotent - safe to rerun.

Repeat for each environment:

```powershell
.\output\scripts\setup-sql-identity.ps1 -Environment staging
.\output\scripts\setup-sql-identity.ps1 -Environment prod
```

### Step 5: Deploy Application

Trigger the **Deploy Application** workflow from GitHub Actions:

1. Go to **Actions** -> **Deploy Application** -> **Run workflow**
2. Choose the target environment
3. Click **Run workflow**

The workflow:

- Sets up .NET 10 and restores NuGet packages (cached)
- Publishes `ApiApp` to `./publish/api`
- Publishes `WebApp` to `./publish/web`
- Logs in to Azure via OIDC
- Deploys API App to `api-accessdbapp-{env}` via `azure/webapps-deploy@v3`
- Deploys Web App to `app-accessdbapp-{env}` via `azure/webapps-deploy@v3`

**Note**: EF Core migrations are applied automatically at API App startup via `DbInitializer` in `Program.cs`. No separate migration step is needed.

The **Deploy Application** workflow also runs automatically after a successful **Build and Test** run on the `main` branch (deploys to `dev`).

## Continuous Integration

The **Build and Test** workflow (`.github/workflows/build-app.yml`) runs on every push and pull request to `main`:

1. Checkout code
2. Setup .NET 10
3. Cache NuGet packages
4. `dotnet restore AppModBoosterV2x.sln`
5. `dotnet build AppModBoosterV2x.sln --configuration Release --no-restore`
6. `dotnet test output/test/unit/ApiApp.Tests.csproj` (72 unit tests, NUnit 4, in-process SQLite)
7. Upload `.trx` test results as artifact (retained 14 days)

E2E tests (Playwright) are not run in CI - they require both apps to be running.

## Environment-Specific Configuration

### API App (`output/src/ApiApp/`)

| File                           | Environment   | Key Settings                                                                               |
| ------------------------------ | ------------- | ------------------------------------------------------------------------------------------ |
| `appsettings.json`             | All           | `ConnectionStrings:DefaultConnection` (empty; overridden per env), `Swagger:Enabled: true` |
| `appsettings.Development.json` | Local dev     | SQLite connection string (`northwind-local.sqlite`), Swagger enabled                       |
| `appsettings.dev.json`         | Azure dev     | SQL Server connection string with Managed Identity auth, Swagger enabled                   |
| `appsettings.staging.json`     | Azure staging | SQL Server connection string with Managed Identity auth, Swagger enabled                   |
| `appsettings.prod.json`        | Azure prod    | SQL Server connection string with Managed Identity auth, Swagger disabled                  |

The `ASPNETCORE_ENVIRONMENT` app setting on each App Service controls which `appsettings.{env}.json` is loaded.

### Web App (`output/src/WebApp/`)

| File                           | Environment   | Key Settings                                            |
| ------------------------------ | ------------- | ------------------------------------------------------- |
| `appsettings.json`             | All           | `ApiSettings:BaseUrl: http://localhost:5201`            |
| `appsettings.Development.json` | Local dev     | API base URL: `http://localhost:5201`                   |
| `appsettings.dev.json`         | Azure dev     | API base URL set to `api-accessdbapp-dev` Azure URL     |
| `appsettings.staging.json`     | Azure staging | API base URL set to `api-accessdbapp-staging` Azure URL |
| `appsettings.prod.json`        | Azure prod    | API base URL set to `api-accessdbapp-prod` Azure URL    |

The `ApiSettings__BaseUrl` app setting is injected into the Web App by the Bicep module (`webApp.bicep`) automatically during infrastructure deployment.

## Teardown

To delete all Azure resources for an environment, trigger the **Teardown Infrastructure** workflow:

1. Go to **Actions** -> **Teardown Infrastructure** -> **Run workflow**
2. Select the environment to delete
3. Type the environment name again in the **confirm** field (e.g., `dev`)
4. Click **Run workflow**

The workflow deletes the entire resource group `rg-accessdbapp-{env}` and all resources within it. This action is irreversible.

> **Warning**: Tearing down `prod` will delete the production database and all data. Ensure backups exist before proceeding.
