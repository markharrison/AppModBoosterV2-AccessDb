# Deployment Guide

> Generated by booster-techwriter | 2026-04-07 15:58:31

## Prerequisites

Before deploying to Azure, ensure the following are available:

- **Azure subscription** with Contributor access
- **Azure CLI** (`az`) installed and authenticated (`az login`)
- **PowerShell 7+** (scripts use PowerShell, not Bash)
- **.NET 10 SDK** installed
- **GitHub repository** with Actions enabled
- **sqlcmd** with Azure AD authentication support (for database access grants)

## Infrastructure Overview

All resources are provisioned using Azure CLI PowerShell scripts — no Bicep or Terraform is used. Scripts are idempotent and safe to rerun after partial completion.

Resources per environment (naming: `{resourcetype}-{app}-{env}`, default app name: `northwind`):

| Resource | Name Pattern | Purpose |
| --- | --- | --- |
| Resource Group | `rg-{app}-{env}` | Container for all resources |
| App Service Plan | `plan-{app}-{env}` | Linux, B1 default (overridable), shared by both apps |
| API App | `app-{app}-api-{env}` | ASP.NET Core Web API, managed identity, health check |
| Web App | `app-{app}-web-{env}` | ASP.NET Core Razor MVC, managed identity |
| SQL Server | `sql-{app}-{env}` | Azure AD-only auth, TLS 1.2, no SQL auth |
| SQL Database | `sqldb-{app}-{env}` | Basic tier, 2 GB max |
| Application Insights | `appi-{app}-{env}` | Telemetry — `prod` only |
| Log Analytics | `log-{app}-{env}` | Backing App Insights — `prod` only |

## Step 1: Set Up OIDC Identity for GitHub Actions

Run the OIDC identity setup script to create an Entra ID app registration with federated credentials for all three environments:

```powershell
.\output\scripts\setup-oidc-identity.ps1 -GitHubOrg <your-org> -GitHubRepo <your-repo>
```

**Parameters:**

| Parameter | Required | Default | Description |
| --- | --- | --- | --- |
| `-GitHubOrg` | Yes | — | GitHub organization or user name |
| `-GitHubRepo` | Yes | — | GitHub repository name |
| `-AppName` | No | `northwind` | Application name prefix |
| `-Location` | No | `eastus2` | Azure region |

**What it does:**
1. Creates an Entra ID app registration named `gh-{appname}-deployer`
2. Creates a service principal
3. Creates OIDC federated credentials for `dev`, `staging`, and `prod` environments
4. Assigns the Contributor role on the subscription

**After running**, configure these GitHub repository secrets (values are displayed in the script output):

| Secret Name | Value |
| --- | --- |
| `AZURE_CLIENT_ID` | Application (client) ID of the app registration |
| `AZURE_TENANT_ID` | Azure AD tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Azure subscription ID |

**Create these GitHub Environments** (Settings → Environments):
- `dev`
- `staging`
- `prod`

## Step 2: Provision Infrastructure

### Option A: Via GitHub Actions (recommended)

1. Go to **Actions** → **Deploy Infrastructure**
2. Click **Run workflow**
3. Select the target environment (`dev`, `staging`, or `prod`)
4. Optionally override: Azure region (default: `eastus2`), App Service SKU (default: `B1`), app name (default: `northwind`)
5. Click **Run workflow**

The workflow authenticates via OIDC, runs `deploy-infra.ps1`, and then runs `grant-db-access.ps1`.

### Option B: Run locally

```powershell
# Provision infrastructure
.\output\infra\deploy-infra.ps1 -Environment dev

# Grant database access to the API App managed identity
.\output\infra\grant-db-access.ps1 -Environment dev
```

**`deploy-infra.ps1` parameters:**

| Parameter | Required | Default | Description |
| --- | --- | --- | --- |
| `-Environment` | Yes | — | `dev`, `staging`, or `prod` |
| `-Location` | No | `eastus2` | Azure region |
| `-AppServiceSku` | No | `B1` | App Service Plan SKU |
| `-AppName` | No | `northwind` | Application name prefix |

**What `deploy-infra.ps1` does:**
1. Creates resource group
2. Creates App Service Plan (Linux)
3. Creates API App and Web App with managed identity, HTTPS-only, TLS 1.2
4. Configures app settings (connection strings, environment variables, ports)
5. Creates SQL Server with Azure AD-only authentication
6. Creates SQL Database (Basic tier, 2 GB)
7. Creates Application Insights + Log Analytics (`prod` only)

**`grant-db-access.ps1` parameters:**

| Parameter | Required | Default | Description |
| --- | --- | --- | --- |
| `-Environment` | Yes | — | `dev`, `staging`, or `prod` |
| `-AppName` | No | `northwind` | Application name prefix |

**What `grant-db-access.ps1` does:**
- Acquires an Azure AD access token for SQL
- Connects to the SQL Database using `sqlcmd`
- Creates a database user for the API App managed identity
- Grants `db_datareader`, `db_datawriter`, and `db_ddladmin` roles

## Step 3: Deploy the Application

### Option A: Via GitHub Actions (recommended)

1. Go to **Actions** → **Deploy Application**
2. Click **Run workflow**
3. Select the target environment
4. Click **Run workflow**

The workflow:
1. Builds both API App and Web App (.NET 10, Release configuration)
2. Runs the 87 unit tests
3. Publishes both apps as artifacts
4. Authenticates to Azure via OIDC
5. Deploys API App to `app-{app}-api-{env}` using `azure/webapps-deploy@v3`
6. Deploys Web App to `app-{app}-web-{env}` using `azure/webapps-deploy@v3`
7. Verifies API health by polling `/health` (up to 10 retries, 15s apart)

### Option B: Deploy manually

```powershell
# Build and publish
dotnet publish output/src/ApiApp/ApiApp.csproj --configuration Release --output ./publish/api
dotnet publish output/src/WebApp/WebApp.csproj --configuration Release --output ./publish/web

# Deploy using Azure CLI
az webapp deploy --name app-northwind-api-dev --resource-group rg-northwind-dev --src-path ./publish/api
az webapp deploy --name app-northwind-web-dev --resource-group rg-northwind-dev --src-path ./publish/web
```

## CI/CD Workflows

All workflows are in `.github/workflows/` and use OIDC authentication (no stored Azure credentials).

| Workflow | File | Trigger | Purpose |
| --- | --- | --- | --- |
| **Build Application** | `build-app.yml` | Push/PR to `main` (paths: `output/src/**`, `output/test/unit/**`), manual | Build both apps, run 87 unit tests, upload artifacts |
| **Deploy Application** | `deploy-app.yml` | Manual (`workflow_dispatch`) | Build, test, deploy both apps to selected environment |
| **Deploy Infrastructure** | `deploy-infra.yml` | Manual (`workflow_dispatch`) | Provision Azure resources for selected environment |
| **Delete Infrastructure** | `delete-infra.yml` | Manual (`workflow_dispatch`) | Tear down Azure resources for selected environment |

### Workflow Authentication

All workflows use OIDC federated credentials:

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - uses: azure/login@v2
    with:
      client-id: ${{ secrets.AZURE_CLIENT_ID }}
      tenant-id: ${{ secrets.AZURE_TENANT_ID }}
      subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
```

### GitHub Actions Used

## Application Configuration (Cloud)

### API App Settings

Set automatically by `deploy-infra.ps1`:

| Setting | Value |
| --- | --- |
| `ASPNETCORE_ENVIRONMENT` | `dev` / `staging` / `prod` |
| `DatabaseProvider` | `SqlServer` |
| `ConnectionStrings__DefaultConnection` | `Server=sql-{app}-{env}.database.windows.net;Database=sqldb-{app}-{env};Authentication=Active Directory Managed Identity;TrustServerCertificate=True;` |
| `WEBSITES_PORT` | `5201` |
| `APPLICATIONINSIGHTS_CONNECTION_STRING` | Set for `prod` only |

### Web App Settings

Set automatically by `deploy-infra.ps1`:

| Setting | Value |
| --- | --- |
| `ASPNETCORE_ENVIRONMENT` | `dev` / `staging` / `prod` |
| `ApiApp__BaseUrl` | `https://app-{app}-api-{env}.azurewebsites.net` |
| `WEBSITES_PORT` | `5200` |
| `APPLICATIONINSIGHTS_CONNECTION_STRING` | Set for `prod` only |

## Environment Teardown

### Option A: Via GitHub Actions

1. Go to **Actions** → **Delete Infrastructure**
2. Click **Run workflow**
3. Select the environment to delete
4. Click **Run workflow**

The workflow runs `delete-infra.ps1` with `-Force` (skips confirmation prompt).

### Option B: Run locally

```powershell
# With confirmation prompt
.\output\infra\delete-infra.ps1 -Environment dev

# Skip confirmation
.\output\infra\delete-infra.ps1 -Environment dev -Force
```

This deletes the entire resource group and all contained resources. The deletion runs asynchronously in Azure.

## Deployment Checklist

- [ ] OIDC identity created (`setup-oidc-identity.ps1`)
- [ ] GitHub secrets configured (`AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_SUBSCRIPTION_ID`)
- [ ] GitHub environments created (`dev`, `staging`, `prod`)
- [ ] Infrastructure provisioned (`deploy-infra.ps1` or Deploy Infrastructure workflow)
- [ ] Database access granted (`grant-db-access.ps1`)
- [ ] Application deployed (Deploy Application workflow)
- [ ] API health check passing (`https://app-{app}-api-{env}.azurewebsites.net/health`)
- [ ] Web App accessible (`https://app-{app}-web-{env}.azurewebsites.net`)
