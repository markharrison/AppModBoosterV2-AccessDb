# Architecture - Northwind Traders

## System Overview

Northwind Traders is a cloud-native web application modernized from a Microsoft Access database (`AccessDBApp.accdb`) to a fully managed Azure PaaS solution. The system consists of two ASP.NET Core applications - a RESTful API backend and a Razor MVC frontend - backed by Azure SQL Database and monitored via Azure Application Insights.

The application exposes standard CRUD operations for customers, employees, orders, products, and order details, along with reporting and admin capabilities inherited from the original Access application.

## Authentication and Authorization

### Local Development

No authentication is configured for local development. Both applications run without identity checks, enabling rapid iteration against a seeded SQLite database.

| Component | Auth | Database                          |
| --------- | ---- | --------------------------------- |
| Web App   | None | n/a (calls API)                   |
| API App   | None | SQLite (`northwind-local.sqlite`) |

### Cloud (dev / staging / prod)

All cloud-to-cloud communication uses **Azure Managed Identity** - no passwords, connection strings with credentials, or API keys are stored anywhere.

| Connection              | Authentication Method                                                   |
| ----------------------- | ----------------------------------------------------------------------- |
| Web App -> API App      | HTTP (same App Service Plan, private)                                   |
| API App -> Azure SQL    | `Authentication=Active Directory Managed Identity` (system-assigned MI) |
| GitHub Actions -> Azure | OIDC federated credentials (no `AZURE_CREDENTIALS` secret)              |

#### Managed Identity Flow (API App -> SQL)

```
API App (system-assigned MI)
  |
  | Token request (IMDS endpoint)
  v
Azure AD
  |
  | Short-lived access token
  v
Azure SQL Database
  (contained user: api-accessdbapp-{env})
```

The API App's managed identity is created as a contained database user at deploy time via `setup-sql-identity.ps1`. Permissions granted: `db_datareader`, `db_datawriter`, `db_ddladmin`.

## Infrastructure Summary

All resources follow the naming convention `{type}-accessdbapp-{env}` where `{env}` is `dev`, `staging`, or `prod`.

| Resource                | Name Pattern              | SKU / Tier          | Notes                                   |
| ----------------------- | ------------------------- | ------------------- | --------------------------------------- |
| Resource Group          | `rg-accessdbapp-{env}`    | n/a                 | Container for all env resources         |
| App Service Plan        | `plan-accessdbapp-{env}`  | Linux B1            | Shared by Web App and API App           |
| API App (App Service)   | `api-accessdbapp-{env}`   | -                   | System-assigned MI, HTTPS-only, TLS 1.2 |
| Web App (App Service)   | `app-accessdbapp-{env}`   | -                   | System-assigned MI, HTTPS-only, TLS 1.2 |
| SQL Server              | `sql-accessdbapp-{env}`   | -                   | Entra-only auth, no SQL login, TLS 1.2  |
| SQL Database            | `sqldb-accessdbapp-{env}` | Basic (5 DTU, 2 GB) | Collation: SQL_Latin1_General_CP1_CI_AS |
| Application Insights    | `appi-accessdbapp-{env}`  | Workspace-based     | Backed by Log Analytics                 |
| Log Analytics Workspace | `log-accessdbapp-{env}`   | PerGB2018           | 30-day retention                        |

Infrastructure is defined as Bicep in `output/infra/`:

```
output/infra/
  main.bicep                    # Top-level orchestrator
  modules/
    appServicePlan.bicep        # Shared Linux App Service Plan
    apiApp.bicep                # API App + managed identity + SQL connection
    webApp.bicep                # Web App + managed identity + API base URL
    sqlServer.bicep             # SQL Server + Database, Entra-only
    monitoring.bicep            # Application Insights + Log Analytics
  parameters/
    dev.bicepparam              # Dev environment parameters
    staging.bicepparam          # Staging environment parameters
    prod.bicepparam             # Prod environment parameters
```

## API Surface

The API App exposes 8 controller groups under `http://localhost:5201` (local) or `https://api-accessdbapp-{env}.azurewebsites.net` (cloud):

| Controller               | Base Route          | Operations                                   |
| ------------------------ | ------------------- | -------------------------------------------- |
| `CustomersController`    | `/api/customers`    | CRUD + paged list                            |
| `EmployeesController`    | `/api/employees`    | CRUD + paged list                            |
| `OrdersController`       | `/api/orders`       | CRUD + paged list                            |
| `OrderDetailsController` | `/api/orderdetails` | CRUD by order                                |
| `ProductsController`     | `/api/products`     | CRUD + paged list                            |
| `AdminController`        | `/api/admin`        | Settings, seed, cleanup                      |
| `ReportsController`      | `/api/reports`      | Sales by employee/product, email/phone lists |
| `LookupController`       | `/api/lookup`       | Reference data for dropdowns                 |

Full OpenAPI spec: `output/api/openapi.json`
