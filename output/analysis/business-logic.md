## 1. Core Logic

### 1.1 Application Startup Sequence

The `AutoExec` macro runs on database open. The startup sequence is:

1. Open `frmStartup` (splash/loading screen).
2. Instantiate `g_ErrorHandler` (`clsErrorHandler`) - global error handler.
3. Call `OneTimeProcessing()` (`modGlobal`) if `SystemSettings.FirstTimeProcessingSuccess` is `False`:
   a. `SetDatesToCurrent()` - shifts demo order dates to current timeframe.
   b. `SetCtrlCurrencyFormat()` - fixes currency format properties on forms and reports for regional locale.
   c. `AddDataMacros()` - programmatically attaches audit-trail data macros to tables that have the audit columns but no existing macro.
   d. Marks `SystemSettings.FirstTimeProcessingSuccess = -1` (True).
4. Call `SetAppTitle()` to set the Access window title to `"Northwind Traders Starter Edition  2.4"`.
5. Open `frmLogin`.

### 1.2 Authentication

**Mechanism**: Windows identity matched against the `Employees` table. No password stored.

**Logic** (`frmLogin`):

1. Read `Environ("USERNAME")` via `GetWindowsUserName()` (`modGlobal`).
2. Call `DLookup` against `qryEmployeeLogin` WHERE `WindowsUserName = [username]`.
3. If a match is found, store `EmployeeID` in `TempVars!UserID`.
4. If no match is found, default to `DEFAULT_LOGIN_ID = 2` (hardcoded constant in `modGlobal`).
5. `TempVars!UserID` is the session-scoped identity reference used throughout the application.

**Special employee IDs**:

- `DEFAULT_LOGIN_ID = 2`: fallback when Windows username is not recognised.
- `INTERNET_SALES_EMPLOYEEID = 10`: reserved for automated / simulated internet orders created by `CreateRandomOrders()`.

### 1.3 Audit Trail (Data Macros)

All primary business tables (Customers, Employees, Orders, OrderDetails, Products, OrderStatus) contain four audit columns: `AddedBy`, `AddedOn`, `ModifiedBy`, `ModifiedOn`.

These are populated automatically by Access data macros attached to each table's `BeforeChange` event. The macros call `GetAuditFieldsUserName()` (`modTableDataMacro`), which:

- Returns `TempVars!UserID` resolved to `Employees.FullNameFNLN` if the user is logged in.
- Falls back to `GetWindowsUserName()` (raw `Environ("USERNAME")`) if `TempVars!UserID` is null.

Because Access templates cannot include data macros referencing custom VBA functions, `AddDataMacros()` adds the macros programmatically at first run:

1. Reads the macro XML from `Welcome.DataMacro` (stored in the database as a Memo field).
2. Writes the XML to a temp file (`%TEMP%\datamacro.xml`).
3. Calls `Application.LoadFromText acTableDataMacro, tableName, filePath` for each qualifying table.

### 1.4 Order Total Calculation

Order totals are not stored. They are calculated on demand via query `qryOrderTotal`:

```sql
SELECT OrderDetails.OrderID, Sum([Quantity]*[UnitPrice]) AS OrderTotal
FROM OrderDetails
GROUP BY OrderDetails.OrderID
```

This query is referenced by `qryOrders` (for list display) and `qryrptSalesByEmployee` / `qryrptSalesByProduct` (for reports).

The `UnitPrice` captured in `OrderDetails` is a snapshot taken at the time the line item is entered (copied from `Products.UnitPrice` via the combo box default). Product price changes do not retroactively affect existing order line item totals.

### 1.5 Tax Rate

Tax rate is stored in `SystemSettings` (ID=1, Name="TaxRate") as an integer (e.g., `85`). At runtime it is divided by 1000 via `GetSystemSetting()` to produce a `Single` value (`0.085` = 8.5%). This approach avoids decimal/regional format issues when storing numeric values as text.

### 1.6 Order Status Workflow

Status transitions are user-driven via the `StatusID` combo box on `frmOrderDetails`. There is no enforced state-machine constraint in code; users can freely select any status.

The query `qryActiveCustomers` filters customers whose orders have `StatusID <> 1` (i.e., excludes Closed orders).

### 1.7 Sales Summary Queries

Two summary queries produce dashboard metrics:

**`qrySales_YTD`**: Year-to-date revenue total (current calendar year).

```sql
SELECT Sum([Quantity]*[UnitPrice]) AS OrderTotal
FROM Orders INNER JOIN OrderDetails ON Orders.OrderID = OrderDetails.OrderID
WHERE Year([OrderDate]) = Year(Date())
```

**`qrySales_SalesRep`**: Year-to-date revenue per employee.

```sql
SELECT Sum([Quantity]*[UnitPrice]) AS OrderTotal, Employees.FullNameFNLN
FROM (Employees INNER JOIN Orders ON Employees.EmployeeID = Orders.EmployeeID)
INNER JOIN OrderDetails ON Orders.OrderID = OrderDetails.OrderID
WHERE Year([OrderDate]) = Year(Date())
GROUP BY Employees.FullNameFNLN
```

### 1.8 Internet / Simulated Order Creation

`modOrders.CreateRandomOrders(intOrderCount)`:

1. Opens recordsets on `Orders` and `OrderDetails` tables.
2. For each order: inserts an `Orders` record with `EmployeeID = INTERNET_SALES_EMPLOYEEID`, a random `CustomerID`, `OrderDate = Now()`, and `StatusID = osNew (3)`.
3. Creates 2-5 random `OrderDetails` line items per order, each with a random `ProductID`, random `Quantity` (5-50), and `UnitPrice` looked up from `Products` via `DLookup`.
4. On duplicate key error (3022), retries with a new random product ID.
5. Calls `DoCmd.RunMacro "macMainMenu_UpdateSubs"` to refresh UI after completion.

Random selection helpers (`modDAO`):

- `GetRandomPkValue(strTable, strPkField)`: opens a snapshot recordset, moves to a random row, returns the PK value.

### 1.9 Date Reset

`SetDatesToCurrent()` (referenced from `modGlobal.OneTimeProcessing`) shifts all `OrderDate` values proportionally so that the most recent demo order date becomes today. This makes year-to-date queries meaningful against template data.

### 1.10 Error Handling

All modules use a standardised error-handling pattern:

- Each procedure has `On Error GoTo Err_Handler` and a numbered exit label.
- `HandleError(strModuleName, strProcedureName)` on the global `g_ErrorHandler` (`clsErrorHandler`):
  1. Captures `Err.Number`, `Err.Description`, and `Erl` (line number).
  2. Appends a log entry to `Northwind-Starter.log` in the database folder via `LogErrorToFile()` with timestamp, module/procedure, error details, Windows username, and machine name.
  3. Shows a `MsgBox` with the error information.
- Exception: Error 2501 (`OpenForm` was cancelled) is silently ignored.

## 3. Roles and Permissions

### Identity Mechanism

The application uses **Windows-username-based identity** resolved against the `Employees` table. There is no built-in Access workgroup security (Access security groups were removed in .accdb format).

- Identity source: `Environ("USERNAME")` (operating system login name).
- Identity store: `Employees.WindowsUserName` (Text, 50 characters).
- Session token: `TempVars!UserID` (Long Integer = EmployeeID).
- Resolution code: `frmLogin`, `qryEmployeeLogin`, `GetWindowsUserName()`.

### User Roles

No formal role-based access control is defined in the application. All authenticated employees share the same access level. The following functional distinctions exist:

| Role (informal)    | How Identified                                                | Capabilities                                                           |
| ------------------ | ------------------------------------------------------------- | ---------------------------------------------------------------------- |
| Regular employee   | Any `Employees.WindowsUserName` match                         | Full access to all forms: customers, orders, products, reports         |
| Default/guest user | No matching `WindowsUserName`; falls back to `EmployeeID = 2` | Same access as regular employee                                        |
| Internet sales bot | `EmployeeID = 10` (INTERNET_SALES_EMPLOYEEID constant)        | Orders assigned to this ID by `CreateRandomOrders()`; not a real login |

### Access to Admin Functions

`frmAdmin` is not linked from the standard main menu navigation. It must be opened directly (e.g., via `DoCmd.OpenForm "frmAdmin"` from a hidden button or keyboard shortcut). No programmatic permission check guards its opening.

### Audit Identity

The audit fields (`AddedBy`, `ModifiedBy`) record either:

- The resolved `Employees.FullNameFNLN` (if `TempVars!UserID` is set), or
- The raw Windows username (if not logged in).
