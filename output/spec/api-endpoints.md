# API Endpoints Specification

**Source app**: Northwind Traders Starter Edition v2.4 (Access .accdb)
**Target**: ASP.NET Core Web API, Controller-based, .NET 10

---

## Overview

All routes are prefixed with `/api`. The API App is accessed server-side only by the Web App via HttpClient. Swagger UI is enabled in local dev and dev/staging environments; disabled in production.

### Standard Response Conventions

| Scenario            | HTTP Status                                 |
| ------------------- | ------------------------------------------- |
| Success (with body) | `200 OK`                                    |
| Created             | `201 Created` + `Location` header           |
| Success (no body)   | `204 No Content`                            |
| Not found           | `404 Not Found` + `{ "error": "..." }`      |
| Validation failure  | `400 Bad Request` + `{ "errors": { ... } }` |
| Server error        | `500 Internal Server Error`                 |

### Shared Query Parameters

Where listed for collection endpoints:

- `page` (int, default 1) - 1-based page number
- `pageSize` (int, default 20, max 100) - items per page

### GET /api/orders

List all orders with rich filtering. Mirrors `sfrmMainMenu_OrderList` and the four `qryOrders_MostRecent_*` queries.

**Query parameters**:
| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `customerId` | int | No | Filter by customer |
| `employeeId` | int | No | Filter by employee |
| `status` | string | No | Filter by status name (e.g., `New`, `Closed`) |
| `sortBy` | string | No | `orderDate` (default), `modifiedOn` - controls "most recent" mode |
| `top` | int | No | Returns only the top N records (for side panels - legacy top-20 queries) |
| `page` | int | No | |
| `pageSize` | int | No | |

**Response body** `200 OK`:

```json
{
  "items": [
    {
      "id": 1,
      "customerId": 1,
      "customerName": "...",
      "employeeId": 2,
      "employeeFullName": "...",
      "orderDate": "2026-01-15T00:00:00Z",
      "shippedDate": null,
      "paidDate": null,
      "status": "New",
      "statusDisplay": "New",
      "orderTotal": 245.0,
      "modifiedOn": "2026-01-15T00:00:00Z"
    }
  ],
  "totalCount": 200,
  "page": 1,
  "pageSize": 20
}
```

**Business rules**:

- `orderTotal` is calculated as `SUM(Quantity * UnitPrice)` across `OrderDetails` for each order.
- When `top` is provided, pagination parameters are ignored.

### GET /api/products

List all products with optional search and sales summary statistics.

**Query parameters**:
| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|

| `pageSize` | int | No | |

{

      "id": 1,

**Business rules**:

- `soldThisWeek` = count of distinct order IDs that contain this product where `OrderDate` falls in the current ISO calendar week.
- Stats fields are `null` when `includeSalesStats=false`.

## Assumptions

1. **No authentication enforcement at endpoint level in local dev**: All endpoints are accessible without tokens in the `Development` environment. Cloud environments enforce Managed Identity between Web App and API App.

2. **Audit field population**: The API App determines the current user identity for audit fields from request context (e.g., the calling user's identity claim or a service header). In local dev, a placeholder identity (e.g., `"dev-user"`) is used. This mechanism is architectural detail for the implementation agent.

3. **Admin endpoint security**: `DELETE /api/admin/test-data` and `POST /api/admin/generate-orders` should be protected in cloud environments. The mechanism (role claim, specific managed identity, etc.) is deferred to the implementation agent.

4. **No server-side paging on lookup endpoints**: Lookup endpoints return all records without paging, as they back dropdown controls that need the full list. For large datasets, this may need revisiting.

5. **Invoice tax always applied**: The invoice endpoint always applies the `TaxRate` system setting. If tax-exempt customers are required in future, a `Customer.IsTaxExempt` field would be needed.

6. **Status filter format**: The `status` query parameter on `GET /api/orders` accepts the enum name string (e.g., `New`, `Closed`), not the numeric value, for readability.

7. **Order notes - plain text only**: The API stores and returns notes as plain text. No RTF handling is implemented in the new app.

8. **`top` parameter vs pagination**: When `top` is specified on collection endpoints, it is not composable with `page`/`pageSize`. The top-N behavior replicates the legacy "most recent 20" side-panel query pattern.
