# API Endpoints

REST endpoint catalog for the modernized Northwind Traders API App. All endpoints follow controller-based REST conventions on ASP.NET Core Web API. Derived from the CRUD matrix in `./output/analysis/business-logic.md` and UI workflows.

Base path: `/api`

Standard HTTP status codes used throughout:

| Code | Meaning                    |
| ---- | -------------------------- |
| 200  | OK                         |
| 201  | Created                    |
| 204  | No Content (delete/update) |
| 400  | Bad Request                |
| 404  | Not Found                  |
| 500  | Internal Server Error      |

---

Creates a new customer. Sets `AddedBy` and `AddedOn` audit fields automatically.

**Request body:**

```json

  "customerName": "Contoso Ltd",
  "primaryContactFirstName": "Jane",
  "primaryContactLastName": "Doe",

  "state": "WA",
```

### PUT /api/customers/{id}

Updates an existing customer. Sets `ModifiedBy` and `ModifiedOn` audit fields automatically.

**Path parameters:** `id` (int)
**Request body:** Same shape as POST (without `customerId`)
**Response 200:** Updated customer object
**Response 404:** Customer not found

## Assumptions

- There is no user authentication between the Web App and the API App in local development. In cloud environments, managed identity is used (as defined in `copilot-instructions.md`). The API endpoints do not implement per-endpoint authorization checks beyond what the managed identity boundary provides.
- All endpoints return JSON. Content negotiation is not required.
- Admin endpoints (`/api/admin/*`) are not protected by a separate role-claim in local development. In a future iteration, role-based access can be added; however, the legacy app had no programmatic access control on the admin panel either.
- The `generate-orders` endpoint uses `EmployeeId = 10` (INTERNET_SALES_EMPLOYEEID) for seeded orders, consistent with legacy behavior. Employee ID 10 must be present in seed data.
- Date range filters (`from`, `to`) on report endpoints accept ISO 8601 date strings (yyyy-MM-dd).
- `orderTotal` on order list/detail responses is computed server-side (sum of line items) and not stored in the database.
