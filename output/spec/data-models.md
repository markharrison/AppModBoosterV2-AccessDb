# Data Models Specification

**Source app**: Northwind Traders Starter Edition v2.4 (Access .accdb)
**Target**: EF Core code-first, SQL Server (cloud) / SQLite (local dev)

---

## Overview

Five core entities are carried forward from the legacy database. Two lookup tables (`NorthwindFeatures`, `Welcome`) are omitted as they contain help/onboarding content with no business value in the modernized app. `OrderStatus` is represented as a C# enum to remove a mutable reference table that carries fixed domain data. `SystemSettings` is retained in a simplified form for the `TaxRate` setting.

---

## Audit Base Class

All core entities inherit a shared audit base to replicate the Access data-macro behavior.

```
AuditableEntity (abstract)
  AddedBy     : string?   -- display name of creating user
  AddedOn     : DateTime? -- UTC timestamp of creation
  ModifiedBy  : string?   -- display name of last modifier
  ModifiedOn  : DateTime? -- UTC timestamp of last modification
```

The API App will populate these fields server-side via EF Core `SaveChanges` interceptor or override, using the authenticated identity (local dev: a placeholder; cloud: Managed Identity resolved employee name).

## OrderStatus Enum

Maps from: `OrderStatus` table (fixed 5-row lookup)

```csharp
public enum OrderStatus
{
    New       = 3,  // StatusCode: NEW — initial state
    Invoiced  = 2,  // StatusCode: INV
    Shipped   = 4,  // StatusCode: SHI
    Paid      = 5,  // StatusCode: PAY
    Closed    = 1,  // StatusCode: CLO
}
```

Enum integer values intentionally match legacy `StatusID` values to preserve data compatibility on migration.

**Display order** (for dropdowns/labels, from legacy `SortOrder`):

1. New (3)
2. Invoiced (2)
3. Shipped (4)
4. Paid (5)
5. Closed (1)

**Business rule**: In the new app, status transitions are free-form (no machine-enforced state transitions), matching legacy behavior.

## Database Seeding

The API App seeds the database on startup in non-production environments using EF Core's `HasData` or a custom seed service. Seeded data includes:

- Sample Customers (from original Access data)
- Sample Employees (including `Id=2` default user and `Id=10` internet sales bot)
- Sample Products (with ProductCode and UnitPrice)
- Sample Orders and OrderDetails (with dates shifted to be current-year-relevant, mirroring legacy `SetDatesToCurrent()` behavior)
- `SystemSettings` row for `TaxRate`
- All `OrderStatus` enum values documented above (no DB table needed once enum is used)

## Assumptions

1. **Attachments field dropped**: The Access `Employees.Attachments` multi-value attachment field has no direct EF Core / SQL Server equivalent. File attachment capability is out of scope for this modernization. The field is dropped. If file attachments are required in future, a separate `EmployeeAttachment` entity and Azure Blob Storage integration should be designed separately.

2. **RTF Notes**: The legacy `Notes` field on `Orders` was stored as RTF (Access Memo field with `PlainText()` stripping used for display). The new app stores plain text only. Any migrated data should be stripped of RTF markup before import.

3. **WindowsUserName retained**: `Employee.WindowsUserName` is retained in the data model to support data migration traceability and historical audit field interpretation. It is not used for authentication in the new app.

4. **Audit fields nullable**: All four audit fields (`AddedBy`, `AddedOn`, `ModifiedBy`, `ModifiedOn`) are nullable to allow partial population during data migration.

5. **Decimal precision**: Legacy Access `Currency` type has 4 decimal places. `decimal(18,4)` is used for all monetary values to match this precision.

6. **OrderStatus enum values**: Enum integer values are intentionally set to match the legacy `StatusID` values (non-sequential) to simplify data migration. The stored int column in SQL Server will hold these values.

7. **SystemSettings extensibility**: Only `TaxRate` is ported to the `SystemSettings` table. If additional application configuration is needed, standard `appsettings.json` / environment variables are the preferred mechanism in the new app rather than extending this table.

8. **No soft delete**: The legacy app had an admin `DeleteTestData` bulk-delete function. The new app uses standard hard deletes. Cascading delete rules should be defined on FK relationships (Orders cascade-delete their OrderDetails; Customers/Employees set null on Orders rather than cascade, matching legacy Left Join patterns).
