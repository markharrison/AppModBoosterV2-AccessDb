# Data Models

Defines entities for the modernized Northwind Traders application. All business entities mapped from the legacy Access database. Legacy-only metadata tables (NorthwindFeatures, Welcome) are omitted as they are not required in the new application.

---

## OrderDetail

Maps from legacy `OrderDetails` table.

| Property      | C# Type   | Nullable | Notes                                 |
| ------------- | --------- | -------- | ------------------------------------- |
| OrderDetailId | int       | No       | Primary key, auto-generated           |
| OrderId       | int?      | Yes      | FK to Order.OrderId                   |
| ProductId     | int?      | Yes      | FK to Product.ProductId               |
| Quantity      | short?    | Yes      | Number of units                       |
| UnitPrice     | decimal?  | Yes      | Price snapshot at time of order entry |
| AddedBy       | string?   | Yes      | Max 255; audit - set by application   |
| AddedOn       | DateTime? | Yes      | Audit - set by application            |
| ModifiedBy    | string?   | Yes      | Max 255; audit - set by application   |
| ModifiedOn    | DateTime? | Yes      | Audit - set by application            |

### Relationships

- Belongs to `Order` (via OrderId)
- Belongs to `Product` (via ProductId)

### Computed Properties

- `ExtendedPrice` (read-only): `Quantity * UnitPrice`

### Business Rules

- `UnitPrice` is a snapshot copied from `Product.UnitPrice` at the time the line item is created. Changes to product prices do not retroactively affect existing order details.

## Audit Trail Pattern

All business entities (Customer, Employee, Order, OrderDetail, Product) include four audit fields managed by the application layer on create/update operations:

| Field      | Type      | Set On | Value Source                                 |
| ---------- | --------- | ------ | -------------------------------------------- |
| AddedBy    | string?   | Create | Authenticated employee full name or username |
| AddedOn    | DateTime? | Create | Current UTC timestamp                        |
| ModifiedBy | string?   | Update | Authenticated employee full name or username |
| ModifiedOn | DateTime? | Update | Current UTC timestamp                        |

These are set in the API App service/controller layer before persisting to the database. There are no database triggers.

## Assumptions

- The `Attachments` field on the `Employees` table (Access multi-value attachment type) is dropped. Access attachment fields have no simple equivalent in SQL Server with EF Core. If file attachments are needed in future, a separate file storage mechanism should be designed.
- The `NorthwindFeatures` and `Welcome` tables are omitted. They served as in-application help content in the legacy desktop app and have no functional equivalent in a web application.
- Audit fields are populated by application logic (service/controller layer) rather than database triggers, consistent with EF Core conventions.
- `SystemSettings` will be seeded with the TaxRate setting. The `FirstTimeProcessingSuccess`, `LastResetDate`, and `ShowWelcome` settings were startup-only and are not needed in the new application.
- `UnitPrice` is stored as `decimal` (EF Core maps to `decimal(18,2)` by default) rather than Access Currency type.
- The `WindowsUserName` field on `Employee` is retained for reference but the authentication mechanism in the new app does not depend on it (there is no Windows identity in a web app). The field is kept so migrated data is preserved.
