# UI Pages Specification

**Source app**: Northwind Traders Starter Edition v2.4 (Access .accdb)
**Target**: ASP.NET Core Razor, .NET 10, Bootstrap

---

## Overview

The legacy app had 14 top-level forms, 16 subforms, and 6 reports inside a single Access window. The modernized app maps these to a set of Razor pages organized around 5 resource areas (Orders, Customers, Employees, Products, Reports) plus an Admin section. The Access startup/login/welcome splash forms are replaced by the web app's standard page load behavior.

All pages are server-side rendered Razor pages. The Web App calls the API App via HttpClient. JavaScript is used only for client-side enhancements (dropdowns, row highlighting, inline totals).

### / - Dashboard

**Route**: `/`
**Title**: Dashboard - Northwind Traders
**Legacy source**: `frmMainMenu` (main shell) + `sfrmMainMenu_OrderList` + `sfrmMainMenu_ProductList` + sales summary text boxes

**Purpose**: Application landing page. Provides a summary of key metrics and quick access to recent orders and top products.

**Primary API endpoints consumed**:

- `GET /api/reports/sales-ytd` - YTD total sales figure
- `GET /api/reports/sales-by-employee?year={current}` - per-employee YTD sales summary
- `GET /api/orders?sortBy=orderDate&top=20` - most recent 20 orders
- `GET /api/products?includeSalesStats=true&pageSize=10` - top products with sales stats (for product summary widget)

**Key UI components**:

- **YTD Sales card**: single large total figure (mirrors `qrySales_YTD`)
- **Sales by Rep table/chart**: employee name + YTD total (mirrors `qrySales_SalesRep`)
- **Recent Orders panel**: compact table of the 20 most recent orders with columns: Order #, Date, Customer, Employee, Status, Total. Clicking a row navigates to `/orders/{id}`.
- **Product summary panel**: compact table of products with `QtySold` and `SoldThisWeek` columns.

**Design notes**:

- The YTD summary and per-rep summary should be visually prominent (Bootstrap cards / stat widgets).
- The recent orders table is the primary focus area (matches the legacy default tab).
- No filter controls on the dashboard itself; filtering is done on the Orders list page.

  - Clickable rows or a View link -> `/orders/{id}`
  - Status badge colored by status (e.g., New=blue, Closed=grey) - mirrors `macOrderDetails_SetColor`
- **Pagination** controls (if more than one page)
- **New Order** button (top right) -> `/orders/new`

**Breadcrumb**: Dashboard > Orders

**Legacy source**: `frmOrderDetails` (new record mode)

**Purpose**: Create a new order header. After creation, the user is redirected to `/orders/{id}` to add line items.

**Primary API endpoints consumed**:

### /orders/{id} - Order Detail

**Route**: `/orders/{id:int}`
**Title**: Order #{id} - Northwind Traders
**Legacy source**: `frmOrderDetails` (edit mode) + `sfrmOrderDetails` (line items) + `sfrmOrders_MostRecent` (side panel)

**Purpose**: View and edit a single order including all line items. Print/view invoice.

**Primary API endpoints consumed**:

- `GET /api/orders/{id}` - load order + line items
- `GET /api/lookup/customers`
- `GET /api/lookup/employees`
- `GET /api/lookup/order-statuses`
- `GET /api/lookup/products` - for add line item product dropdown
- `PUT /api/orders/{id}` - save order header changes
- `PATCH /api/orders/{id}/status` - quick status update
- `POST /api/orders/{id}/details` - add line item
- `PUT /api/orders/{id}/details/{lineId}` - edit line item
- `DELETE /api/orders/{id}/details/{lineId}` - remove line item
- `DELETE /api/orders/{id}` - delete order
- `GET /api/orders?customerId={x}&top=20` - recent orders for same customer (side panel)

**Key UI components**:

- **Order header section**: editable fields - Customer (dropdown), Employee (dropdown), Order Date, Shipped Date, Paid Date, Status (dropdown or quick-status buttons), Notes
- **Line items table** (editable inline or via modal):
  - Columns: Product (dropdown), Quantity, Unit Price, Extended Price (calculated client-side), Delete button
  - Add Line Item row / button at the bottom
  - Running total displayed below the table (mirrors `qryOrderTotal`)
- **Side panel**: "Recent Orders" compact list filtered to the same customer - top 20 by date. Clicking a row navigates to that order. (Mirrors `sfrmOrders_MostRecent`)
- **Action bar**: Save, Delete Order, Print Invoice (-> `/reports/invoice/{id}` in new tab), Back to Orders

**Design notes**:

- The inline editable line items table is a significant UI pattern improvement over the Access datasheet subform. Use JavaScript to compute `ExtendedPrice` client-side on quantity/price change.
- Status can be updated inline without saving the full header (via `PATCH /api/orders/{id}/status`).
- The side panel (recent orders for same customer) is a legacy feature worth retaining for workflow efficiency.

**Legacy pattern flag**: The Access form used a locked `PlainText()` helper to strip RTF from Notes display. In the new app, plain text is stored; no special handling needed.

**Breadcrumb**: Dashboard > Orders > Order #{id}

## Assumptions

1. **No login page in local dev**: The legacy app authenticated via Windows username. The new app has no authentication in local dev. A dedicated login page is not specified. If end-user auth (e.g., Microsoft Entra ID OIDC) is added later, a login redirect page would be needed.

2. **No per-user access control in initial spec**: The legacy app had one effective access level for all employees. The new app also has one access level in this spec. Admin panel access restriction in cloud environments is noted but the mechanism is deferred to the implementation agent.

3. **Inline editing vs. separate edit pages**: Some detail pages (Customer, Employee, Product) present the same page in view/edit mode (single Razor page with a form). This matches the legacy single-form-per-record pattern. An alternative is separate `/edit` routes; this spec uses the single-page approach.

4. **Navigation bar Admin link**: The Admin link is always shown in the nav bar. In cloud environments, if the user lacks admin role, the page should return `403 Forbidden` rather than hiding the link.

5. **No "next/prev record" navigation**: The legacy Access forms had prev/next navigation buttons to cycle through records. This pattern is not natural in web UIs and is replaced by returning to the list page.

6. **No welcome/about modal**: `frmWelcome` and `frmAbout` are not ported. An `/about` page can be added optionally but is out of scope for this spec.

7. **Print reports via browser**: All report pages use browser native print (`window.print()` + `@media print` CSS). No server-side PDF generation is specified.

8. **Bootstrap version**: Bootstrap 5.x (latest stable as of target implementation). No integrity attributes on CDN `<script>` / `<link>` tags per `copilot-instructions.md`.

9. **Dashboard sales chart**: The `qrySales_SalesRep` data on the dashboard is specified as a table widget. An optional bar chart (using a lightweight JS library) may be added by the implementation agent, but is not required by this spec.
